# MTX Vibe Coding 任务复盘

## 我做了什么

我完成了 **DROP//DESTINY**：一个以 Bass 音色塑造为核心的浏览器音乐创作游戏。用户先确定声音世界、Bass 性格和节奏骨架，再进入简化合成器亲手调节音色。系统会根据真实参数推导编曲倾向和最终风格，播放六种可能结局之一，并把用户的主 Bass 实时叠加到完整歌曲的 Drop 中。

## 最大的收获

我最大的收获，是学会了让多个 AI Agent 分工协作并持续迭代同一个项目。这也是一次比较完整的小项目经历，让我对规格、开发、测试、素材许可、部署和最终打包都更熟悉了。

我不会从零手写这样规模的代码，但可以明确产品方向，把任务拆给 AI，试玩它的结果，指出具体问题，再要求它修正。技术门槛确实降低了，但“做什么”和“什么才算做好”仍然需要人来决定。

## 最困难的部分

最困难的是音频和网页外观设计。早期音频有明显的占位感：风格之间鼓组差异不够、Bass 和其他声部音量失衡，Build-up 与后半段鼓组也不自然。视觉方面，AI 初版很容易堆很多卡片、边框和常见的紫绿渐变，但这并不是我想要的 Bass Music 气质。

我通过安装更适合音乐编曲、音频生成和可视化的 skill，查找真实合成器与网页作为参考，并不断给出“哪一个声部太小”“哪一段节奏乱”“哪一种颜色不对”这种可执行反馈来解决。最终音乐改为六套预渲染伴奏加用户实时主 Bass，视觉也统一成红、黑、白为主的粗野主义海报风格。

## 一次关键转向

项目最初有较多选择题，还有 Pattern 和 Arrangement 操作。但后来发现这些操作并没有真正改变最终歌曲，继续保留只会制造“假装在创作”的感觉。因此我决定删掉它们，把主要交互集中到 Bass Forge：让用户调真实的振荡器、FM、失真、滤波器、包络、LFO、Sub 与空间参数，同时让 Drive 联动可视化颜色。这个转向让玩法更诚实，也更符合最初“用户自己捏 Bass”的核心创意。

## 一次典型问题与修复

结局伴奏升级后，用户主 Bass 曾经没有在正确的小节进入和结束。原因是实时 Bass 调度与 Collider 伴奏对段落时间的理解不一致。修复时先统一 14 小节时间轴，再让主 Bass 只覆盖 Drop A 与 Drop B，最后为六种风格增加集成测试，验证首音和尾音都落在正确窗口。这个问题让我意识到：听起来像音乐的问题，也需要用清晰的数据结构和自动测试保证。

## 我对 Vibe Coding 的理解

AI 很大地改变了产品开发。它能从自然语言开始编程、测试、优化代码，也能连接外部平台完成部署；人与 Agent 的协作体验也比以前完整得多。但人的作用不是消失了，而是转向了方向判断、体验反馈、审美选择和验收。特别是美术与音乐听感，人仍需起主要作用。

如果重新做一次，我会更早确定唯一的核心交互，更早建立统一的音乐时间轴、自动测试和素材许可证清单，也会少花时间扩展那些最终不会影响作品的功能。
